\section{Results}
\subsection{Selecting Model Architecture and Dataset}

As discussed earlier, preliminary experiments were first carried out to determine the model family and training dataset to be used for the majority of the study. These experiments compared candidate object detection architectures in terms of analytical performance and latency, and also compared several labelled football broadcast datasets. Figure~\ref{fig:arch_compare} compares the analytical performance and latency of a range of object detection model families, while Figure~\ref{fig:dataset_compare} shows the preliminary dataset comparison carried out using the SoccerNet \cite{tracking}, Roboflow \cite{roboflow_sports_repo}, and Football Analytics \cite{roboflow_football_analytics_gnykk_model1} datasets.

As can be seen from Figure \ref{fig:arch_compare}, although Faster R-CNN \cite{Zhao2023RTDETR} and SSD achieved slightly higher initial accuracy, the YOLO family \cite{UltralyticsYOLO} provided substantially lower latency on the Jetson Orin Nano Dev Kit. As a result, YOLO-based models were selected for the subsequent object detection experiments, as they offered the most suitable balance between accuracy and real-time edge deployment performance. Figure~\ref{fig:dataset_compare} indicates that the Football Analytics dataset achieved the strongest initial $\mathrm{mAP}_{50\text{--}95}$, at approximately 91\%, and was therefore selected for the majority of the object detection study.



\subsection{Summary of Model Training}



With the model family and dataset selected, the full training and compression study was then carried out. In total, 904 unique model artefacts were generated and evaluated across the object detection and pitch keypoint tasks. This included 232 \textbf{YOLO11} models, 236 \textbf{YOLO11-pose} models, 204 \textbf{YOLO26} models, and 232 \textbf{YOLO26-pose} models. The majority of this consisted of compressed variants rather than baseline models, reflecting the central aim of the study: to examine how different compression techniques affect performance on the edge device. 

\subsection{Compression Analysis Summary}
Before individually exploring each compression technique and desired hardware resource reduction, it is useful to first summarise the overall trends across the full model inventory. The following tables and accompanying trade-off plots provide a high-level overview of how each compression technique and combination affected the mAP$_{50\text{--}95}$ versus latency relationship for the object detection and pitch keypoint models. These summaries allow the general patterns to be identified before delving into the more detailed per-technique analyses that follow.

\subsubsection{Compression Summaries: Object Models}
Table \ref{tab:object_technique_summary} below displays the average change in mAP50-95 and latency for each of the main optimisation techniques applied to the YOLO11 and YOLO26 object detection models, averaged across the paired N, S, M, L, and X variants.

\input{tables/object_technique_summary}

Hardware acceleration can immediately be seen to be the most effective technique, reducing latency by \textbf{31.0\%} with only a \textbf{-0.3\%} change in mAP$_{50\text{--}95}$. FP16 quantisation also offers a significant latency improvement with little accuracy loss, delivering a \textbf{42.8\%} latency improvement for only a \textbf{-1.6\%} mAP$_{50\text{--}95}$ reduction. INT8 pushes latency lower at a clearer analytical cost. Input reduction produces a more incremental trade-off: 75\% gives a similar latency gain to acceleration, with a larger analytical penalty, while 60\% and 50\% produce a much steeper trade-off. Finally, pruning provides only a small latency gain until the most aggressive setting. 

While knowledge distillation appears to have a negative impact on latency in this summary, this is likely an (irrelevant) artefact of slightly inefficient PyTorch implementations by me. This inefficeincy is negated when distillation is combined with hardware acceleration, which removes the PyTorch dependency. The combination of these various techniques can now be explored, as shown in Table \ref{tab:object_combo_summary}. 


\input{tables/object_combo_summary}

As can be seen in Table \ref{tab:object_combo_summary} and as expected, there is little downside to hardware acceleration. By contrast, distillation, pruning, and a combination of the two introduce trade-offs, with increases in latency and reductions in mAP$_{50\text{--}95}$ that become more pronounced as the compression path becomes more aggressive. The most extreme combined paths, which include INT8 quantisation, achieve the largest latency improvements but also the largest mAP$_{50\text{--}95}$ reductions. The FP16-based combined paths occupy a more balanced middle ground, providing substantial latency improvements while preserving much more of the analytical performance than the INT8 routes. Overall, these summary tables indicate that hardware acceleration is a clear first step for improving deployability, while more aggressive compression techniques require careful consideration of the accuracy--latency trade-off. Note that as expected, the combination of hardware acceleration and knowledge distillation has removed the PyTorch dependency and therefore the latency increase observed for distillation alone.


With a summary of the object detection compression results, the optimisation of pitch keypoint models can now be examined in the same way.
\subsubsection{Keypoint Detection Model Compression Summary}
\noindent Figure \ref{tab:pose_technique_summary} display the same summary view for the pitch keypoint models produced, averaging the effect of each base technique across the paired N, S, M, L, and X variants of the YOLO11 and YOLO26 families.

\input{tables/pose_technique_summary}

Figure \ref{tab:pose_technique_summary} shows that the same techniques that were effective for object detection also produce the strongest latency improvements for keypoint estimation, with hardware acceleration and quantisation displaying little downsides. However, the mAP$_{50\text{--}95}$ reductions are generally more severe for the pose models than for the object models, especially for input resolution reduction and pruning. This indicates that the keypoint task is more sensitive to aggressive compression. As with before, the knowledge distillation latency increase is an effect of poor PyTorch implementation, and is negated when combined with hardware acceleration. Table \ref{tab:pose_combo_summary} and its companion trade-off plot extend this comparison to the YOLO26-pose combined compression paths.

\input{tables/pose_combo_summary}

As can be seen, hardware acceleration and FP16 quantization offer very little downside regardless of the combination of compression techniques. Input reduction causes a much stronger mAP$_{50\text{--}95}$ reduction then with object models, and INT8 again causes a singificant mAP$_{50\text{--}95}$ reduction. With a broad analysis of the effect of compression techniques on both object and pitch keypoint models, a more in-depth examination of each technique can now be carried out, exploring the effect of model size on compression efficiency.

\subsection{Individual Compression Technique Analysis}

\subsubsection{Hardware Acceleration}
Figure \ref{fig:hardware_acceleration_results} presents the mAP50-95 versus latency relationship for the uncompressed and FP32-accelerated YOLO11 and YOLO26 object and pose model families. This plot allows the effect of hardware acceleration on both accuracy and latency to be visualised across all model families and sizes.

\begin{figure}[H]
    \centering
    \caption{Hardware acceleration trade-off by architecture and task.}
    \label{fig:hardware_acceleration_results}
    \includegraphics[width=.9\linewidth]{../Figures/produced_images/map50_95_vs_latency_baseline_pairs.png}
\end{figure}

Figure \ref{fig:hardware_acceleration_results} displays that across almmost all model families and sizes, hardware acceleration produces a significant reduction in latency with almost no change in mAP50-95. The points for the accelerated models are consistently shifted leftward relative to their uncompressed baselines, indicating faster inference, while remaining at approximately the same vertical level, indicating preserved accuracy. The exception to this is the smaller models (YOLO11n, YOLO26n, YOLO26n-pose), where little latency drop is found, and a more significant loss in mAP$_{50\text{--}95}$ is observed. This suggests that the smallest models may already be operating close to a hardware limits of the Jetson Orin Nano, and therefore see less benefit from acceleration. Overall, the figure indicates that hardware acceleration is an effective technique for improving inference speed with minimal impact on analytical performance, especially for the larger model scales.


\subsubsection{Input Resolution Reduction}


Figure \ref{fig:input_resolution_reduction_results} shows the effect of reducing the input resolution on the mAP$_{50\text{--}95}$ versus latency trade-off for the YOLO11 and YOLO26 object and pose models. Each line traces the progression of a baseline model through increasingly aggressive input resolution reduction, allowing the effect of this technique to be examined in a structured way across both model scale and task type.
\begin{figure}[H]
    \centering
    \caption{Input resolution reduction trade-off.}
    \label{fig:input_resolution_reduction_results}
    \includegraphics[width=.9\linewidth]{../Figures/produced_images/map50_95_vs_latency_input_resolution_panels.png}
\end{figure}

As seen, reducing the input resolution produces a significant reduction in latency for all four model families. The object detection models preserve relatively high mAP$_{50\text{--}95}$ values alongside this latency reduction, especially for the lighter input reductions. By contrast, the pose models, and especially the YOLO26 pose variants, experience substantially larger degradations in accuracy as resolution decreases. The exception to this is the YOLO26n-pose model, which maintains a beneficial trade-off between latency and accuracy as input resolution is reduced. 


\subsubsection{Quantisation}
Figure \ref{fig:quantisation_results} shows the average mAP$_{50\text{--}95}$ versus latency trade-off for baseline, FP32, and INT8 deployment stages across the YOLO11 and YOLO26 object and pose families. As with before. each line traces the progression of a specific baseline model through FP16 and INT8 quantisation, allowing the effect on both inference speed and accuracy retention to be examined in a structured way.

\begin{figure}[H]
    \centering
    \caption{Quantisation trade-off.}
    \label{fig:quantisation_results}
    \includegraphics[width=.9\linewidth]{../Figures/produced_images/map50_95_vs_latency_quant_pairs.png}
\end{figure}

As can be seen, FP16 quantisation produces a significant reduction in latency with only a small reduction in mAP$_{50\text{--}95}$ for both object and pose models. The INT8 quantisation stage pushes latency even lower, but at a much more significant cost to accuracy, especially for the pose models. This suggests that while quantisation is an effective technique for improving inference speed, the choice of quantisation level is crucial for maintaining analytical performance, with FP16 offering a more balanced trade-off than INT8 for this specific application and hardware context.

\subsubsection{Pruning}
Figure \ref{fig:pruning_results} illustrates the mA$_{50\text{--}95}$ versus latency relationship for YOLO11 object and pose models under increasing pruning strength, again separated across specific model baselines. Each line traces the progression of a baseline model through 90\%, 80\%, and 70\% pruning.

\begin{figure}[H]
    \centering
    \caption{Pruning trade-off.}
    \label{fig:pruning_results}
    \includegraphics[width=.9\linewidth]{images/map50_95_vs_latency_pruning_pairs_nsmlx.png}
\end{figure}


The object detection results show that pruning generally reduces latency while causing only moderate reductions in mA$_{50\text{--}95}$, particularly for the larger model scales. In several cases, the pruned object models remain close to the baseline in accuracy while offering meaningful speed improvements, which suggests that object detection models retain a degree of redundancy that pruning can exploit. In contrast, the pose models are much more sensitive to pruning. While mild pruning can still provide a useful trade-off, heavier pruning leads to severe reductions in mAP$_{50\text{--}95}$ for most pitch models. This indicates that the keypoint models are less robust to node removal.

\subsubsection{Knowledge Distillation}
The final compression technique to be explored is knowledge distillation. Figure \ref{fig:knowledge_distillation_results} presents the effect of knowledge distillation on each baseline model. Each line traces the progression of a baseline model through small, medium, and large teacher jumps, with a jump size defined by the gap to the teacher model (for example, N to S would be small, N to M would be medium, and N to L would be large). In this analysis, accelerated distillations are compared to accelerated baselines, to negate the PyTorch inefficiencies discussed earlier.

\begin{figure}[H]
    \centering
    \caption{Knowledge distillation trade-off.}
    \label{fig:knowledge_distillation_results}
    \includegraphics[width=.85\linewidth]{images/map50_95_vs_latency_knowledge_ns_chains.png}
\end{figure}


As can be seen, for object models, knowledge distillation provided little latency improve and only a small reduction in mAP$_{50\text{--}95}$. YOLO26-pose models (which have been more robust to compression in previous tests) have significant mAP$_{50\text{--}95}$ reductions with distillation, with only small latency improvements.  Overall, the results suggest that while knowledge distillation can be a useful technique for improving training time, its effectiveness is highly dependent on the specific model architecture and task, with pose estimation models being more sensitive to this form of compression.

\subsection{Best Models}

With compression techniques and their effectiveness analysed, the optimal object detection and pose estimation models can now be selected. Given a target system frame rate of approximately 20 FPS, the combined latency budget for both models sits at around 40--50 ms. Accounting for system-level optimisations such as inference intervals and parallelisation, this budget can reasonably be extended to approximately 80 ms. Within this constraint, the latency results from the preceding sections are used to identify the strongest individual models that remain within budget while preserving as much of the baseline mAP$_{50\text{--}95}$ as possible. Restricting the search to object detection models with a latency under $\frac{80}{2} = 40$ ms yields the plot below, with the top-performing models labelled. The plot is further filtered to mAP$_{50\text{--}95}$ values above 0.8, focusing attention on the most competitive candidates.

\begin{figure}[H]
    \centering
    \caption{Best object detection models under 40 ms latency.}
    \label{fig:best_object_models}
    \includegraphics[width=\linewidth]{../Figures/produced_images/best_model_analysis_object.png}
\end{figure}

As can be seen by Figure \ref{fig:best_object_models}, a clear frotier can be determined. The best performing models with-in the latency budget of 40ms is clearly the FP16 quantised 26s model, with a distilled version, a pruned version, and an INT8 version also performing well. As latency decreases, 11n FP16 also provides a strong option, with a distilled version and pruned version indicating a strong trade-off. Finally, as latency decreases further, the 26s model but with a reduced input resolution of 60\% also provides a strong option, with the pruned version of this model also performing well. It was chosen to not explore models beyond 80\% mAP$_{50\text{--}95}$, as this would likely be too far from the baseline to be useful in the full-stack system. 

With the best object detection models identified, the same process can now be applied to the pose estimation models. This is displayed in Figure \ref{fig:best_pose_models},with again a latency limit of 40ms. For this plot, a limit of 97\% mAP$_{50\text{--}95}$ was used, as drop-off quickly occurs beyond this point without major accuracy imrpovment.

As can be seen by Figure \ref{fig:best_pose_models}, the frontier of pitch keypoint estimation is much less defined. In this case, only 4 models provide optimal latency-accuracy trade-offs, all of which are derivativions of the 26n family. The FP16 quantised 26n model provides the highest accuracy below 40ms, with distilled and pruned versions with reduced input sizes offering faster inference times with slight accuracy loss.

\subsection{Full-Stack System}

With the 9 best performing object models and 3 best peforming keypoint models identified, the full-stack system can be prepared and evaluated, before for accuracy and for hardware efficiency. First, the system itself will be briefly described, before the full-stack results are presented and analysed. 

\subsection{Full-Stack System Implementation}
Figure \ref{fig:fullstack_parameter_screen_2} shows the parameter screen for the full-stack system. This screen allows the user to select the desired video and object and pose models, and select team colour definitions for each teams out-field and in-field players. Various system-level parameters can also be selected here, including the inference intervals, confidence thresholds, pass/possession hyper-parameter settings, and logging level. 

\begin{figure}[H]
    \centering
    \caption{Full-stack system parameter screen.}
    \label{fig:fullstack_parameter_screen_2}
    \includegraphics[width=\linewidth]{images/Screenshot_2026-04-07_152040.png}
\end{figure}

Inference begins on upload with the video is treated as a continuous stream, with frames being read in and processed sequentially. The output of the system is a exposed endpoint containing object and keypoint locations (both in on-screen and 2D pitch coordinates), as well as frame-by-frame pass and possession predictions. The system also contains a HTML visualisation endpoint which, if accessed, will display the video with the detected objects and keypoints overlaid, as well as the predicted passes and possessions. This endpoint is displayed below in Figure \ref{fig:fullstack_parameter_screen_1}.

\begin{figure}[H]
    \centering
    \caption{Full-stack system parameter screen.}
    \label{fig:fullstack_parameter_screen_1}
    \includegraphics[width=\linewidth]{images/Screenshot_2026-04-07_164923.png}
\end{figure}

As the system is not designed to include visualisation, the visualisation endpoint is not optimised for real-time performance and was largely ignored from optimisation. The main key optimisation applied here was the use of client-side rendering rather then the initial server-side rendering, which slightly reduced the latency of the visualisation endpoint by around 50\%.  With the system implemented, the full-stack results can now be presented and analysed.

\subsubsection{Full-Stack System Performance}


These shortlisted models listed in Figures \ref{fig:best_object_models} and \ref{fig:best_keypoint_models} were evaluated in the proposed full-stack pipeline to analyse higher-level statistic accuracy and compared to the prior discussed baseline SoccerNet system, which achieved a pass mAP$_{1\text{--}5}$ of 71.2\% \cite{tracking}, with mAP$_{1\text{--}5}$ defined earlier. Each model combination was run on a continuous 10 minute football video within the system defined earlier (with the visualisation endpoint disabled). 

The most key analysis is the relationship between latency and Pass mAP$_{1\text{--}5}$. Figure \ref{fig:sweep_new_map1_5_vs_latency} shows the pass mAP$_{1\text{--}5}$ versus latency for the top 15 combinations.


\begin{figure}[H]
    \centering
    \caption{Pass mAP$_{1\text{--}5}$ versus Latency.}
    \label{fig:sweep_new_map1_5_vs_latency}
    \includegraphics[width=\linewidth]{../Figures/produced_images/sweep_new_map1_5_vs_latency.png}

As can be seen by Figure \ref{fig:sweep_new_map1_5_vs_latency}, the best performing model combinations get quite close to the SoccerNetbaseline pass mAP$_{1\text{--}5}$ of 71.2\%. This is a strong result, indicating that the optimised models can achieve near-baseline analytical performance while operating within the latency budget for real-time deployment. At default, these models operate within a latency bound of 80-110 ms, equivalent to around 9-12 FPS, which is below the target of 20-25 FPS. However, with system-level optimisations such as parallelization and inference intervals, it is likely that the effective latency can be reduced further to meet the real-time requirements.

Figure \ref{fig:piechart_latency_breakdown} shows the percentage breakdown of two implementations of the fullstack system, with the left chart utilising un-compressed models and the right chart utilising two compressed models.

As can be seen by Figure \ref{fig:piechart_latency_breakdown}, keypoint esimation found the great relative reduction in latency, moving from \textbf{23.5\%} of the total latency to \textbf{11.2\%}.
